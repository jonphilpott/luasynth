LuaSynth: Architecture & Lua Embedding Tutorial

A deep dive into how LuaSynth works — from the SDL2 audio callback thread to Lua userdata metatables, lock-free ring buffers, and the pull-model signal graph. This document teaches you how to build a scriptable C application using Lua as a runtime scripting engine.

What Is This Tutorial?

LuaSynth is deliberately small. Its purpose is not to be the world's most sophisticated synthesizer — it is to be a clear, well-commented demonstration of how to embed the Lua scripting language inside a real-time C application that also has to manage audio hardware.

By the end of this tutorial you will understand:

  • What it means to "embed" Lua and why you might want to do it
  • How the four architectural layers of LuaSynth fit together
  • The pull-model signal graph that powers real-time audio
  • How to expose C objects to Lua using full userdata and metatables
  • How to store Lua callbacks in C and call them safely
  • How two threads (main + SDL audio) share state without data races
  • How a lock-free SPSC ring buffer passes events from the audio thread to Lua
Prerequisites
You should be comfortable with C (pointers, structs, function pointers). Some familiarity with Lua syntax is helpful but not required — the Lua snippets are simple enough to follow cold. No DSP experience is necessary; the audio concepts are explained as they appear.

What Is Lua Embedding?

Lua is often described as a "scripting language designed to be embedded." That phrase deserves unpacking, because it explains every architectural decision in this project.

Two ways to use a scripting language

Most scripting languages run as standalone interpreters: you type python myscript.py and the Python runtime owns the process. The C code lives inside the Python interpreter, not the other way around.

Embedding flips this relationship. Your C application is the host. It creates a Lua interpreter as a library object — a lua_State* — and loads scripts into it at runtime. The C code controls when scripts run, what functions they can call, and what data they can access. Lua becomes a safe, sandboxed scripting layer that can configure and control the C application without being able to crash it (or at least, without being able to do anything you didn't give it permission to do).

Why embed Lua specifically?

Lua was designed from day one for embedding. The entire Lua runtime is around 300 KB of compiled code. It has no global mutable state (everything lives in a lua_State), so you can run multiple independent interpreters in the same process. The C API is clean, stack-based, and thoroughly documented. Its garbage collector is incremental. And Lua's syntax is expressive enough to write real music composition logic while remaining easy enough to learn in an afternoon.

The pattern LuaSynth uses

LuaSynth follows the standard embedded-Lua pattern used by many production applications (game engines, text editors, network servers):

  1. C creates a lua_State and loads the standard libraries
  2. C registers custom functions and objects (the "bindings") into the Lua global table
  3. C runs a user-supplied .lua script
  4. The script uses the registered objects to configure the application
  5. The script returns; the application runs indefinitely, driven by the configuration
  6. When things happen (beats fire, parameters change), C calls back into Lua

In LuaSynth the "application" is a modular synthesizer and the "configuration" is a signal patch — a directed acyclic graph of audio modules wired together.

The Four Layers

LuaSynth has four architectural layers that sit cleanly on top of each other. Understanding where each layer lives, and what it owns, is the key to understanding the whole system.

The arrows between layers are strictly controlled. The script layer only talks to C via the binding functions we explicitly register. The binding layer is the gatekeeper: it validates arguments, acquires locks before modifying audio parameters, and mediates all cross-thread communication.

Startup Sequence

src/main.c performs every initialisation step in dependency order, then runs the event loop forever. Here is the sequence:

  1. Initialise SDL2. SDL_Init(SDL_INIT_AUDIO) loads the audio subsystem. Nothing else (no video, no events) is needed.
  2. Create the signal graph. signal_graph_init() zeroes the graph struct. It is empty — no modules yet. Lua will populate it.
  3. Create the ring buffer. Used to pass clock beat events from the audio thread to the main thread. Initialised before the audio device so there is no race.
  4. Open the audio device. audio_engine_init() calls SDL_OpenAudioDevice() and starts the SDL audio callback thread. From this point the callback fires every ~11.6 ms, pulling from the (still-empty) signal graph and outputting silence.
  5. Initialise the Lua engine. Creates a lua_State, loads all standard libraries, stores the engine pointer in the Lua registry, then calls each lua_XXX_register() function to install the module bindings as Lua globals.
  6. Run the Lua script. luaL_dofile() parses and executes the user's patch script. The script runs synchronously on the main thread. When it returns, the patch is wired and the audio thread is producing real sound.
  7. Enter the event loop. The main thread polls SDL events (for quit), calls lua_engine_poll() to drain beat events and fire Lua callbacks, then sleeps 1 ms. This continues until the user presses Ctrl-C. 
/* From main.c — the event loop */
while (running) {
    /* 1. Handle Ctrl-C / window close */
    while (SDL_PollEvent(&event)) {
        if (event.type == SDL_QUIT) running = 0;
    }

    /* 2. Drain clock beat events → call Lua callback functions */
    lua_engine_poll(&lua_engine);

    /* 3. Yield the CPU so we don't spin at 100% */
    SDL_Delay(1);
}

The Signal Graph

The pull model

Audio graphs can work in two directions. A push model has sources generate samples and push them downstream through the graph. A pull model has the output sink ask its upstream neighbours for samples, which in turn ask their neighbours — recursively, up the graph — until everyone has computed what they need.

LuaSynth uses the pull model, which is the standard choice for low-latency audio. The SDL audio callback fires and asks the signal graph for N frames. The graph calls output_module->process(). The output module calls process() on each of its inputs, which in turn call their inputs, and so on. The recursion bottoms out at leaf nodes (oscillators, noise generators) that synthesise samples from scratch.

Pull direction (requests) ←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←← Osc → Filter → Delay → Output → SDL callback ↑ LFO (mod input) →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→ Data direction (samples)

The Module struct

Every source and processor in the graph is a Module — a single C struct defined in src/module.h. It uses the composition pattern: a void *data pointer holds the module-specific state (oscillator phase, filter coefficients, etc.) and two function pointers (process, destroy) act as a lightweight vtable. 

struct Module {
    ModuleType   type;          /* OSC, FILTER, LFO, etc. */

    /* Audio-rate signal connections (upstream producers) */
    Module      *inputs[MODULE_MAX_INPUTS];
    int          num_inputs;

    /* Control-rate modulation connections (LFOs → params) */
    ModInput     mod_inputs[MODULE_MAX_MOD_INPUTS];
    int          num_mod_inputs;

    /* Virtual dispatch — set once at creation time */
    int  (*process)(Module *self, float *out_buf, int frames, int rate);
    void (*destroy)(Module *self);

    void        *data;         /* OscData*, FilterData*, etc. */
    int          in_graph;     /* 1 = owned by the signal graph */
    float        scratch[MODULE_SCRATCH_SAMPLES]; /* pre-allocated temp buffer */
};

The scratch buffer deserves attention. The audio callback is a real-time context: you must never allocate memory inside it (malloc can block on a mutex inside the OS allocator). By pre-allocating a scratch buffer inside every module struct, each module always has a place to temporarily store an upstream module's output without calling malloc.

Connecting modules

module_connect(dst, src) simply adds src to dst->inputs[]. When dst->process() runs, it iterates inputs[], calling each upstream module's process() into self->scratch, then adds those samples into its own output buffer — achieving additive mixing of audio-rate signals.

Control-rate modulation (LFOs sweeping filter cutoff) uses a separate mod_inputs[] array. The target module's process() reads the modulator's output per-sample and adds it (scaled by depth) to the base parameter value before computing its own output.

The Audio Engine

src/audio_engine.c wraps SDL2's audio API. The core idea is simple: we tell SDL what format we want (SDL_AUDIO_F32, 44100 Hz, stereo, 512 frames per buffer) and give it a callback function pointer. SDL creates a dedicated OS thread and calls our callback whenever the hardware needs more audio data. 

/* SDL calls this on its audio thread every ~11.6 ms */
static void sdl_audio_callback(void *userdata, Uint8 *stream, int len) {
    AudioEngine *engine = (AudioEngine *)userdata;

    /* How many stereo frames fit in the byte buffer?
     * Each frame = 2 channels × 4 bytes (float32) = 8 bytes */
    int bytes_per_frame = AUDIO_CHANNELS * (int)sizeof(float);
    int num_frames      = len / bytes_per_frame;

    /* Pull samples from the signal graph into SDL's buffer */
    signal_graph_process(engine->graph, (float *)stream, num_frames,
                         engine->sample_rate);
}
Real-time constraint
The audio callback must return within ~11.6 ms (at 44100 Hz / 512 frames). If it takes longer, SDL's buffer underruns and you hear glitches. This is why the callback cannot allocate memory, cannot acquire non-trivial locks, and cannot call Lua. All those operations have unbounded latency.

The lock functions audio_engine_lock() / audio_engine_unlock() wrap SDL_LockAudioDevice() / SDL_UnlockAudioDevice(). Calling SDL_LockAudioDevice() from the main thread blocks until the current audio callback invocation finishes, then pauses the callback from starting again until SDL_UnlockAudioDevice() is called. This gives the main thread a safe window to write parameters that the audio thread reads.

The Event Loop

After the Lua script runs, main.c enters a while(running) loop that does three things per iteration:

  1. Poll SDL events — catches window-close and Ctrl-C on some platforms
  2. Call lua_engine_poll() — drains clock beat events and fires Lua callbacks
  3. Sleep 1 ms — yields the CPU rather than busy-spinning

This loop is the main thread's entire runtime job. All the actual audio work happens on SDL's audio thread. The main thread only wakes up to handle external events (beats, quit) and to call back into Lua.

Creating the Lua State

The Lua interpreter lives in a lua_State* — an opaque struct that holds everything: the bytecode VM, the garbage collector, the global table, the call stack, and all loaded libraries. You create one with luaL_newstate() and destroy it with lua_close(). 

/* From lua_engine.c */
engine->L = luaL_newstate();    /* create the interpreter */
luaL_openlibs(engine->L);        /* load print, math, string, table, … */

After luaL_openlibs(), the Lua environment has all standard library functions available. Then we call our own registration functions to add the LuaSynth-specific globals: 

lua_osc_register(engine->L);      /* installs global "Osc" table */
lua_filter_register(engine->L);   /* installs global "Filter" table */
lua_clock_register(engine->L);    /* installs global "Clock" table */
/* … and so on for every module type */

Once all the registration calls complete, the Lua script runs in an environment where Osc.new(), Filter.new(), Output.set(), etc. are all available as ordinary Lua global variables containing tables with callable functions.

Registering C Functions

Every function the Lua script can call is a C function with this exact signature: 

typedef int (*lua_CFunction)(lua_State *L);

The integer return value is the number of return values left on the Lua stack. Arguments come in from the script on the Lua stack; return values go back out on the stack. The stack is 1-indexed from the bottom; negative indices count from the top.

The Lua stack

The Lua C API is entirely stack-based. When Lua calls your C function, it has already pushed all the arguments. You read them with lua_toXXX() / luaL_checkXXX(), do your work, push return values, and return the count: 

/* A simple C function that adds two numbers */
static int c_add(lua_State *L) {
    double a = luaL_checknumber(L, 1);  /* arg 1 from Lua */
    double b = luaL_checknumber(L, 2);  /* arg 2 from Lua */
    lua_pushnumber(L, a + b);            /* push result */
    return 1;                             /* 1 return value */
}

/* Register it as a global in Lua */
lua_pushcfunction(L, c_add);
lua_setglobal(L, "add");
/* Now Lua can call: local result = add(3, 4) */

The luaL_checkXXX functions raise a Lua error (via longjmp) if the argument has the wrong type. This is intentional: it gives Lua scripts a clean error message rather than a C crash.

Registering a library table

Rather than registering lots of individual globals, LuaSynth groups functions into tables. luaL_newlib() creates a table and populates it from a luaL_Reg[] array: 

static const luaL_Reg osc_lib[] = {
    {"new", lua_osc_new},
    {NULL, NULL}            /* sentinel */
};
luaL_newlib(L, osc_lib);   /* creates table {new = lua_osc_new} */
lua_setglobal(L, "Osc");  /* Lua: Osc = { new = <C function> } */
/* Now Lua can call: local o = Osc.new("sine", 440) */

Full Userdata Pattern

This is the core technique of LuaSynth's Lua/C integration, and the most important thing to understand. We need Lua scripts to hold references to C objects (oscillators, filters, etc.) and call methods on them. Lua's mechanism for this is full userdata.

What is full userdata?

A "full userdata" is a Lua-managed block of raw memory of a specified size. Lua allocates it on its heap, the garbage collector tracks it, and you can attach a metatable to it (giving it methods). From the C side it looks like a void*.

We use full userdata to hold a pointer to our C module structs — not the struct itself. This is the crucial detail: we allocate a block of exactly sizeof(Module*) bytes inside Lua's heap, then write our Module* pointer into it. 

/* lua_osc_new — Osc.new("sine", 440) in Lua calls this C function */
static int lua_osc_new(lua_State *L) {
    const char *wave_str = luaL_optstring(L, 1, "sine");
    double freq          = luaL_optnumber(L, 2, 440.0);

    /* 1. Allocate the C module struct */
    Module *mod = osc_create(osc_waveform_from_string(wave_str), freq);

    /* 2. Allocate a pointer-sized block in Lua's heap,
     *    write our Module* into it */
    Module **ud = (Module **)lua_newuserdata(L, sizeof(Module *));
    *ud = mod;

    /* 3. Attach the metatable — this gives the userdata its methods */
    luaL_getmetatable(L, "LuaSynth.Osc");
    lua_setmetatable(L, -2);

    return 1;  /* return the userdata to Lua */
}

To retrieve the Module* inside any method function: 

static Module *osc_check(lua_State *L, int idx) {
    /* luaL_checkudata verifies the metatable matches "LuaSynth.Osc".
     * If not, it raises a Lua error with a helpful message. */
    Module **ud = (Module **)luaL_checkudata(L, idx, "LuaSynth.Osc");
    return *ud;  /* dereference the pointer */
}

Why pointer-to-pointer?

Because the C module struct is owned by the signal graph (not by Lua). We want Lua to hold a reference to something, but the lifetime of the actual struct is managed by signal_graph_destroy(), not by Lua's garbage collector. Storing a pointer in Lua's userdata block neatly separates these two lifetimes.

Metatables & OOP in Lua

A metatable is a regular Lua table that controls the behaviour of another table or userdata — it provides operator overloading, method dispatch, and lifecycle hooks (the __gc destructor). This is how Lua achieves object-oriented programming without built-in class syntax.

Setting up the metatable

luaL_newmetatable(L, "LuaSynth.Osc");  /* create and register the metatable */

/* __index = metatable (the OOP trick)
 * When Lua does osc:setFrequency(), it looks for "setFrequency" in the
 * metatable's __index. Setting __index = metatable itself means method
 * lookups search the metatable. */
lua_pushvalue(L, -1);                   /* push copy of metatable */
lua_setfield(L, -2, "__index");        /* metatable.__index = metatable */

/* Register all methods into the metatable */
static const luaL_Reg methods[] = {
    {"setFrequency",  lua_osc_set_frequency},
    {"setAmplitude",  lua_osc_set_amplitude},
    {"setWaveform",   lua_osc_set_waveform},
    {"connect",       lua_osc_connect},
    {"connectMod",    lua_osc_connect_mod},
    {"__gc",          lua_osc_gc},         /* destructor */
    {"__tostring",    lua_osc_tostring},   /* print(osc) */
    {NULL, NULL}
};
luaL_setfuncs(L, methods, 0);

How method calls work

When Lua executes osc:setFrequency(440), it is syntactic sugar for osc.setFrequency(osc, 440). Lua looks up setFrequency on the userdata, finds nothing there, then consults __index — which is the metatable — and finds the C function. It calls the C function with osc as argument 1 and 440 as argument 2.

The __gc destructor

When Lua's garbage collector decides the userdata is unreachable, it calls the __gc metamethod. LuaSynth uses this to free modules that were never added to the signal graph. If a module was added to the graph (mod->in_graph == 1), the graph owns it and we leave it alone: 

static int lua_osc_gc(lua_State *L) {
    Module **ud = (Module **)lua_touserdata(L, 1);
    if (ud && *ud && !(*ud)->in_graph) {
        module_free(*ud);   /* only free if graph doesn't own it */
        *ud = NULL;
    }
    return 0;
}

The Lua Registry

The Lua registry is a special table, accessible from C but invisible to Lua scripts, that lives at pseudo-index LUA_REGISTRYINDEX. It is the standard place to store C-side data that needs to survive across multiple API calls without being a global variable.

Storing the engine pointer

Every binding function (e.g., lua_osc_set_frequency) needs access to the AudioEngine so it can call audio_engine_lock(). We could use a global C variable, but that prevents multiple Lua states from coexisting. Instead we store the LuaEngine* pointer in the registry, keyed by a static variable's address (which is globally unique without being mutable): 

/* A static variable whose ADDRESS is the key — value doesn't matter */
static char engine_registry_key;

/* Store it (called once during init) */
lua_pushlightuserdata(L, &engine_registry_key);  /* key   */
lua_pushlightuserdata(L, engine);                 /* value */
lua_settable(L, LUA_REGISTRYINDEX);

/* Retrieve it (called from every binding function that needs it) */
LuaEngine *lua_engine_get(lua_State *L) {
    lua_pushlightuserdata(L, &engine_registry_key);
    lua_gettable(L, LUA_REGISTRYINDEX);
    LuaEngine *engine = (LuaEngine *)lua_touserdata(L, -1);
    lua_pop(L, 1);
    return engine;
}

This pattern — use a static variable's address as a unique key — is idiomatic embedded-Lua style. It costs nothing at runtime and cannot collide with any other key.

Storing Lua Callbacks in C

When a Lua script calls clk:onBeat(function(beat) ... end), it is passing a Lua function to C. C needs to hold onto that function reference so it can call it later (from lua_engine_poll()). The standard mechanism is luaL_ref(). 

/* clk:onBeat(fn) — store the Lua function for later */
static int lua_clock_on_beat(lua_State *L) {
    clock_check(L, 1);                            /* validate 1st arg = clock */
    luaL_checktype(L, 2, LUA_TFUNCTION);          /* validate 2nd arg = function */

    LuaEngine *eng = lua_engine_get(L);

    /* luaL_ref pops the value at stack top and stores it in the registry,
     * returning an integer key we can use to retrieve it later. */
    lua_pushvalue(L, 2);                           /* copy fn to stack top */
    eng->beat_callback_ref = luaL_ref(L, LUA_REGISTRYINDEX);

    return 0;
}

/* Later, in lua_engine_poll(), to call it: */
lua_rawgeti(L, LUA_REGISTRYINDEX, eng->beat_callback_ref);  /* push fn */
lua_pushinteger(L, beat_number);                               /* push arg */
lua_pcall(L, 1, 0, 0);                                         /* call fn(beat) */

lua_pcall is the protected call. Unlike lua_call, it catches Lua errors (syntax errors, runtime exceptions, explicit error() calls) and returns an error code rather than crashing your C application. Always use lua_pcall when calling back into Lua from a C event loop.

Locks and Atomics

SDL_LockAudioDevice — for parameter writes

When a Lua script calls osc:setFrequency(880), the binding function does this: 

static int lua_osc_set_frequency(lua_State *L) {
    Module    *mod  = osc_check(L, 1);
    double     freq = luaL_checknumber(L, 2);
    LuaEngine *eng  = lua_engine_get(L);

    audio_engine_lock(eng->audio);   /* waits for callback to finish, then pauses it */
    osc_set_frequency(mod, freq);     /* safe to write now */
    audio_engine_unlock(eng->audio); /* lets callback resume */
    return 0;
}

This is slightly heavy — we pause audio for microseconds — but it is simple, correct, and the pause is inaudible because it only happens when the script explicitly changes a parameter (not continuously on every sample).

C11 _Atomic — for flags

For single-word boolean flags like "should the envelope trigger?" or "is the clock running?", a full SDL lock would be overkill. C11's _Atomic int provides atomic load/store with memory-order guarantees, at essentially zero cost: 

/* envelope.h — trigger flag */
typedef struct {
    _Atomic int triggered;   /* main thread writes; audio thread reads */
    /* … */
} EnvelopeData;

/* Main thread (Lua binding): */
atomic_store(&d->triggered, 1);    /* no lock needed */

/* Audio thread (envelope process()): */
if (atomic_load(&d->triggered)) {
    d->phase = ENV_ATTACK;
    atomic_store(&d->triggered, 0);
}
Rule of thumb
Use SDL_LockAudioDevice for multi-byte parameter changes (floats, structs). Use _Atomic for single-word flags and counters that only need to be written or read atomically, not in combination with anything else.

The Ring Buffer

The clock module needs to fire a Lua callback on every beat — but Lua must only be called from the main thread. The audio thread cannot call Lua directly. The solution is a Single-Producer Single-Consumer (SPSC) ring buffer: the audio thread writes beat events, the main thread reads them.

Audio Thread (producer) Main Thread (consumer) clock fires a beat lua_engine_poll() loops │ │ ▼ ▼ ringbuf_push(&ring, beat_event) while (ringbuf_pop(&ring, &ev)) { │ lua_pcall(L, beat_fn, beat); │ } ▼ ┌────┬────┬────┬────┬────┬────┬────┬────┐ │ ev │ ev │ │ │ │ │ │ │ └────┴────┴────┴────┴────┴────┴────┴────┘ ▲write_pos ▲read_pos (audio thread owns) (main thread owns)

The ring buffer is lock-free because each thread only writes its own index: the audio thread advances write_pos, the main thread advances read_pos. C11 atomic_store/atomic_load with acquire/release ordering ensures each thread sees the other's updates correctly without any mutex. 

/* Producer (audio thread) — from ringbuffer.c */
int ringbuf_push(RingBuffer *rb, RingEvent event) {
    int w = atomic_load_explicit(&rb->write_pos, memory_order_relaxed);
    int r = atomic_load_explicit(&rb->read_pos,  memory_order_acquire);
    if ((w - r) >= RINGBUF_CAPACITY) return 0;  /* full — drop */
    rb->slots[w % RINGBUF_CAPACITY] = event;
    atomic_store_explicit(&rb->write_pos, w + 1, memory_order_release);
    return 1;
}

/* Consumer (main thread) */
int ringbuf_pop(RingBuffer *rb, RingEvent *event) {
    int r = atomic_load_explicit(&rb->read_pos,  memory_order_relaxed);
    int w = atomic_load_explicit(&rb->write_pos, memory_order_acquire);
    if (r == w) return 0;  /* empty */
    *event = rb->slots[r % RINGBUF_CAPACITY];
    atomic_store_explicit(&rb->read_pos, r + 1, memory_order_release);
    return 1;
}
Memory ordering
memory_order_release on the write ensures all writes to the slot complete before the index update is visible. memory_order_acquire on the read ensures the consumer sees the slot's data after it sees the updated index. Together they form the classic acquire/release handshake that makes this pattern safe without a mutex.

Walkthrough: Example 01 — Sine Tone

Let us trace every step that happens when you run the simplest example: scripts/examples/01_sine_tone.lua. 

-- 01_sine_tone.lua
local osc = Osc.new("sine", 440.0)
Output.set(osc)

Here is the full execution trace:

  1. Lua evaluates Osc.new("sine", 440.0). Lua looks up the global Osc table (installed by lua_osc_register()), finds new, and calls the C function lua_osc_new() with two arguments on the stack.
  2. C allocates the oscillator. osc_create(OSC_SINE, 440.0) allocates a Module struct and an OscData struct. Sets mod->process = osc_process. Registers it with the signal graph (signal_graph_add()), which sets mod->in_graph = 1.
  3. C creates a Lua userdata. lua_newuserdata(L, sizeof(Module*)) allocates 8 bytes (pointer size) on Lua's heap. The Module* is written into it. The "LuaSynth.Osc" metatable is attached. The userdata is returned to Lua and assigned to osc.
  4. Lua evaluates Output.set(osc). The C function lua_output_set() retrieves the Module* from the userdata and calls signal_graph_set_output(graph, mod). The graph now has an output module.
  5. The Lua script returns. main.c enters the event loop. The SDL audio thread is already running.
  6. SDL calls sdl_audio_callback() every ~11.6 ms. The callback calls signal_graph_process(), which calls output_module->process(). The output module is the oscillator. osc_process() runs: for each of the 512 frames, it advances the phase accumulator by 440/44100, computes sin(phase * 2π), and writes the result into the SDL buffer. The hardware plays a 440 Hz sine tone.
The phase accumulator
The oscillator's frequency is implemented by a phase variable phase ∈ [0, 1) that advances by frequency / sample_rate each sample. When phase reaches 1.0, it wraps back to 0. One full cycle = one complete value range of the waveform. For sine: sin(phase * 2π). For saw: 2*phase - 1.

Walkthrough: Example 04 — Clock & Envelopes

scripts/examples/04_envelope_pluck.lua demonstrates the full cross-thread event path. It is the most architecturally instructive example. 

-- 04_envelope_pluck.lua (abbreviated)
local notes = { 57, 60, 62, 63, 67, 69, 67, 63, 62, 60 }
local beat_index = 1

local osc = Osc.new("square", midi_to_freq(notes[1]))
local env = Envelope.new(0.005, 0.08, 0.2, 0.15)
env:setInput(osc)

local flt = Filter.new("svf")
flt:setInput(env)
Output.set(flt)

local clk = Clock.new(160)
clk:onBeat(function(beat)
    local note = notes[beat_index]
    beat_index = (beat_index % #notes) + 1
    osc:setFrequency(midi_to_freq(note))
    env:trigger()
end)
clk:start()

Here is what happens at each beat, tracing through all four layers:

  1. Audio thread: clock fires. Inside clock_process(), the clock's phase accumulator advances each sample. When it completes a full beat cycle at 160 BPM, it calls ringbuf_push(&ring, beat_event) with the beat number. This is the only thing the clock does — it writes one event to the ring buffer and goes back to accumulating. No Lua is touched.
  2. Main thread: lua_engine_poll() drains the ring buffer. On the next iteration of the while(running) loop (≤1 ms later), ringbuf_pop() returns the beat event. The engine retrieves the stored callback reference from the Lua registry and calls lua_pcall().
  3. Lua callback runs on the main thread. The anonymous function advances beat_index, calls osc:setFrequency() (which acquires the SDL lock, writes the new frequency, releases the lock), then calls env:trigger() (which does an atomic_store to set the trigger flag).
  4. Audio thread: envelope responds. On the very next audio callback invocation, envelope_process() sees the trigger flag set, resets to the attack phase, and begins the ADSR curve. The output is a short plucked note at the new pitch.
Key insight
The beat callback runs on the main thread, never on the audio thread. This is why it is safe to call Lua from the callback — Lua is not thread-safe and must only ever be used from one thread. The ring buffer is the bridge that makes this possible without introducing any mutex between the audio thread and Lua.

How to Add a New Module

To add, say, a bitcrusher distortion module, you would follow this recipe. Every existing module in LuaSynth follows exactly the same pattern, so you can use src/modules/osc.c and src/lua_bindings/lua_osc.c as templates.

Step 1: Define the data struct and C API

Create src/modules/bitcrusher.h and src/modules/bitcrusher.c. The .h declares your data struct and the public functions. The .c implements process(), destroy(), and the factory function: 

/* bitcrusher.h */
typedef struct {
    float  bits;      /* bit depth, e.g. 8.0 */
    float  rate;      /* sample-rate reduction factor */
    float  last;      /* last held sample (for rate reduction) */
    int    counter;
} BitcrusherData;

Module *bitcrusher_create(float bits, float rate);
void    bitcrusher_set_bits(Module *mod, float bits);

Step 2: Implement process()

/* bitcrusher.c — the DSP logic */
static int bitcrusher_process(Module *self, float *out, int frames, int rate) {
    BitcrusherData *d = (BitcrusherData *)self->data;

    /* Pull audio from upstream input */
    if (self->num_inputs > 0) {
        self->inputs[0]->process(self->inputs[0], self->scratch, frames, rate);
    } else {
        memset(self->scratch, 0, frames * 2 * sizeof(float));
    }

    float levels = powf(2.0f, d->bits);
    for (int i = 0; i < frames * 2; i++) {
        /* Quantise: round to nearest 1/levels step */
        out[i] = floorf(self->scratch[i] * levels) / levels;
    }
    return 0;
}

Step 3: Write the Lua binding

Create src/lua_bindings/lua_bitcrusher.c. Copy the structure from lua_osc.c: bitcrusher_check(), lua_bitcrusher_new(), setter functions with audio_engine_lock/unlock, __gc, and lua_bitcrusher_register().

Step 4: Register in lua_engine.c

/* lua_engine.c — add one line to lua_engine_init() */
lua_bitcrusher_register(engine->L);  /* installs global "Bitcrusher" */

Step 5: Add to the Makefile

SRCS += src/modules/bitcrusher.c src/lua_bindings/lua_bitcrusher.c

That is the complete recipe. No other files need to change. The new module is available to Lua scripts immediately: 

local crush = Bitcrusher.new(8.0, 1.0)
crush:setInput(osc)
Output.set(crush)

Key Lessons

Here is a summary of the embedded-Lua patterns that LuaSynth demonstrates, and when to apply each one in your own projects:

Pattern Where in LuaSynth When to use it
Full userdata + metatable
lua_newuserdata + luaL_newmetatable		 Every module binding
lua_osc.c, etc.		 When Lua needs to hold and call methods on a C object. The standard way to do OOP across the Lua/C boundary.
Pointer-to-pointer userdata
Module **ud = lua_newuserdata(L, sizeof(Module*))		 All module bindings When the C object's lifetime is managed outside Lua (e.g., by a graph or pool). Decouples Lua GC from C ownership.
Registry key via static address
static char key; lua_pushlightuserdata(L, &key)		 lua_engine.c When C code needs a unique, stable key for registry storage without a global variable. Supports multiple lua_State instances cleanly.
luaL_ref for callbacks
eng->ref = luaL_ref(L, LUA_REGISTRYINDEX)		 lua_clock.c Whenever a C function needs to store a Lua function for later invocation. Works for any Lua value, not just functions.
lua_pcall not lua_call lua_engine_poll() Always use when calling Lua from a C event loop. Catches errors without crashing your application.
SDL_LockAudioDevice for params Every parameter setter When writing multi-byte values that the audio callback reads. Simplest correct synchronisation; cost is negligible for infrequent updates.
_Atomic for flags envelope.c, clock.c Single-word booleans and counters written from one thread and read from another. Zero overhead, no blocking.
SPSC ring buffer for events ringbuffer.c, clock.c When a realtime thread (audio callback) needs to signal a non-realtime thread (main/Lua) without blocking. Generalises to any producer-consumer event stream.
Pull-model DAG signal_graph.c, all modules Audio processing graphs where you want minimal latency and natural handling of fan-in (multiple inputs summed at a node). Each node computes only when asked.
Pre-allocated scratch buffer module.h, Module.scratch[] Anywhere a realtime callback needs temporary storage. Avoids malloc inside the callback, which would be undefined-latency and is audio death.
